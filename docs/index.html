Un pattern se décrit avec 4 éléments :
Nom : Permet de l'identifier et éventuellement de décrire le problème rapidement
Problématique : Une description du problème que le pattern résoud
Solution : Comment résoudre le problème. Souvent détaillé avec de l'UML
Conséquences : Quels sont les avantages et inconvenients de la solution

Quand ? Pourquoi ?

  • Dès qu'un problème de conception se pose
  • Utilisation d'une solution éprouvée
  • Productivité
  • Qualité de code
  • Bonnes pratiques
  • Over engineering
  • Penser aux Dp dès qu'on est face à un problème de conception
  • Permet d'utiliser une solution sure et éprouvée qui résoud le problème
  • Gain de productivité évident (code déjà fait , connu de tous)
  • Favorise les bonnes pratiques (couplage faible, lisibilité, evolutivité,etc ...)
  • Attention à ne pas en abuser. Peut cacher un problème de conception plus grave. Il préférable de refactorer vers les designs patterns si besoin

  • Patrons structuraux
  • Patrons comportementaux
Créateurs : Comment son créer les objets. => La famille la plus importante

Structuraux et comportementaux sont moins courants => plus souvent rencontré dans le dév de lib ou FW

Les patterns createurs

Singleton

Le plus connu mais aussi le plus controversé. Utilisation très discutable notamment selon les langages

Problématique

  • Une seule et unique instance
  • Accès à cette instance
2 objectifs :
* Une seule instance d'une classe
Important sur gros projets à plusieurs dev => on ne connais pas tout
* Offrir un moyen d'accéder à cette instance

Quand l'utiliser ?

Une seule instance d'une classe peut exister

  • Ressource système
  • Class d'application
  • Gestion de journaux *
* Port série
*
* Fait partie des utilisations très discutable. Des méthodes modernes permettent de s'en passer (DI)

Conséquences

  • Controle l'unicité d'instance d'une classe
  • Induit des problèmes de thread safety
Dans un environnement multi threadé , on risque de casser l'unicité d'instance. Voir plus tard exemple

La controverse

  • Utilité parfois discutable selon le langage
  • Signe de mauvais design
  • Difficile à tester unitairement
* Ok dans les environnement à mémoire partagée. Pas en PHP => instance propre à chaque requête , singleton peut ou pas utile. Suffit de pas instancier 2x la même classe.
* Une sur utilisation du singleton est souvent un signe de mauvais design. On préfère en général l'injection de dépendance.
Une seconde instance va elle crasher mon application => Si non singleton probablement pas nécessaire.
* Créer un couplace fort à l'environnement globale de l'application => rend les tests unitaires plus difficile.

Un peu de théorie

Implémentation naïve

						
public class Singleton {
	private static Singleton instance;

	private Singleton() {}

	public static Singleton getInstance() {
		if (instance == null) {
			instance = new Singleton();
		}
		return instance;
	}
}
- Implémentation fonctionnelle
- Problème de thread safety : - Environnement fortement multi threadé. Peu probable avec 2/3 threads (mais possible)
- plusieurs thread peuvent passer le if en même temps et créer plusieurs instance.

Implémentation Thread Safe

							
public class Singleton {
	private static Singleton instance = new Singleton();

	private Singleton() {}

	public static Singleton getInstance() {
		return instance;
	}
}
On délègue la synchronisation au class loader. Il le gère en interne.
Synchronisation de getInstance possible => /!\ performance
Solution peuvent être différente selon les langages.

Abstract Factory

Problématique

Fournir différents objets d'une même famille sans connaitre leur implémentation concrète

Quand l'utiliser ?

  • Objets regroupés en famille (UI , Documents)
  • Eviter les couplages forts
  • Masquer la complexiter d'instanciation
* Ui : Famille windows/linux; Document : Style moderne/classique * Pas de couplage vers les objets concret. * On expose uniquement les interfaces nécessaires

Fabrique non abstraite

  • Attention à faire la différence. Fabrique = bonne pratique. Pas un DP
  • Abstract factory permet une évolutivité plus simple
* Beaucoup moins de code à changer si nouvelle famille. * Une factory classique va demander plus de modification sur le code qui utilise la factory. * Factory => code figé , Abstract Factory => Code évolutif * Perso : Jamais utilisé d'abstract factory.

Builder

Problématique

Permettre d'obtenir des réprésentations différentes d'un objet avec le même procédé de construction

Séparer la construction d'un objet complexe de sa représentation. On déplace la logique de construction en dehors de l'objet à instancier.

Quand l'utiliser ?

  • Données de construction incomplète
  • Simplification de classe : limiter les constructeurs
* Exemple : parsage d'un document ou réception de données au fur et à mesure. * On évite de surcharger la classe de l'objet avec des infos liées à sa représentation. Ex : des if/else dans le constructeur pour initialiser l'objet. Ou encore plusieurs constructeurs

Un peu de théorie

Présenté pour le principe, mais rarement implémenté sous cette forme trop contraignante.

Solution à 

					
Pizza p = new Pizza();
p.setTaille(Taille.GRANDE);
p.setFromage(Fromage.CHEDDAR);
Ou encore 
					
						Pizza sansolive = new Pizza(Taille.GRANDE,Fromage.CHEDDAR,false);
						Pizza avecjambon = new Pizza(Taille.GRANDE,Fromage.CHEDDAR,false,true);
1- c'est pas très élégant et fastidieux à écrire. Problème si tous les param pas dispo au moment de la construction 2- Compliquer à gérer

Forme la plus courante

						
Pizza pizza = new PizzaBuilder()
	.setTaille(Taille.GRANDE)
	.setFromage(Fromage.MOZZA)
	.setJambon(true)
	.setOlive(false)
	.build();

Les patterns structuraux

Adapter

Problématique

Faire fonctionner ensemble des classes aux interfaces incompatible.
Exemple : Un driver fournis une interface, mais est incompatible avec le besoin du logiciel.
==> On doit adapter son comportement.

Quand l'utiliser ?

  • Besoin de convertir l'interface d'une classe en quelque chose de compréhensible
  • Impossible de modifier l'interface incompatible
Si c'est possible on préfèrera toujours adapter le code.
Exemple : API pilotage => multi caméra mais API constructeur imposé. Il faut adapter la notre.

Decorator

( a.k.a Wrapper )

Problématique

Ajouter des fonctionnalités à une classe dynamiquement.
DYnamique = à l'execution.

Quand l'utiliser ?

  • Ajout de responsabilités optionelles à un objet
  • Pas d'héritage
Responsabilité optionnelle :
=> On doit pouvoir choisir si on utilise ou non les nouvelles responsabilité.

Pas d'héritage :
* Evite l'héritage fonctionnel.
* Manque de souplesse => chaine d'héritage A => B => C , si je veux C je dois tout récupérer.

Exemple concret = la gestion des stream java/C# FileWriter,bufferedWriter , etc ...

Un peu de théorie

  • Component est une interface qu’implémentent à la fois l’objet décoré et les objets qui le décorent
  • ConcreteComponent représente l’objet décoré (qui implémente Component)
  • Decorator représente l’interface des décorateurs
  • On trouve ensuite les décorateurs concrets (ConcreteDecoratorA, etc.)
- L'objet concret et son décorateur hérite de la même interface.
- L'agregation indique qu'un décorateur peut contenir un "component"
- Principale problème => Le type de l'objet sera celui du dernier décorateur utilisé => peut être problématique à gérer.

Pratique

Exemple JAVA dispo
Le but :
* Une classe "Moderator" qui censure du text en remplacant certains mots.
* On ajoute au fur et à mesure la possibilité de censurer un numéro de téléphone puis un email.
La classe Decorator doit être abstraite.

Composite

Problématique

Gérer une hiérarchie d'objet sans limite de profondeur.

Quand l'utiliser ?

  • Hierarchie à profondeur indéterminée
  • Besoin de traiter un parent ou un enfant de manière identique
Chaque élément est un composant potentiel. En plus des éléments classiques, il y a un élément composite qui peut être composé de plusieurs composants. Comme l'élément composite est un composant potentiel, il peut être composé d'autres éléments composites.
L'exemple le plus concret => un système de fichier avec dossier/fichier (chacun peut être cliquer).

Un peu de théorie

Pattern assez naturel , on l'utilise assez souvent sans le savoir.

Facade

Vous l'avez déjà probablement déjà tous utilisé sans le savoir.

Problématique

Simplifier un sous système trop complexe.
Sous système = une partie/module d'un soft
Une API ou une librairie trop complexe

Quand l'utiliser ?

  • Simplifier la lecture/utilisation d'un sous système
  • Refactoring impossible => rétro compatibilité
  • Limiter les dépendances entre le client et le sous système
/!\ A ne pas mélanger avec l'adapteur qui ,lui, doit respecter une interface bien particulière
Facade = vraie volonté de simplification.

Proxy

Problématique

Controler l'accès à un objet en fournissant un intermédiaire
Exemple classique :
* Objet long à répondre (distant, réseau)
* Objet couteux (ram, cpu , etc ...)
On veux alors controller les accès qui sont fait.

Quand l'utiliser ?

  • Accès à l'objet doit être controller
  • Des fonctionnalités supplémentaires sont nécessaires
  • Respect de l'interface de l'objet concret
Exemple concret : Une classe qui affiche une image. On veux charger en mémoire l'image uniquement si elle est affichée et pas avant.
Exemple 2 : Objet distant , on accède à l'objet que quand le réseau est disponible.

Proxies ...

  • Virtual
  • Remote
  • Protection
  • Smart reference
  • Virtuel : retarde l'initialisation et la création d'objet couteux (ex : chargement d'image à l'affichage)
  • Distant : Appel un objet distant (via le réseau par exemple) de manière transparente.
  • Protection : Autorise ou non l'accès à l'objet réel. (ex : gestion de droit)
  • Smart : Accès intelligent à certains objet en comptant leur référence. (ex : smart pointer en C++)

Value Object

Un cas à part

Petit objet qui représente une petite entité dont l'égalité repose sur la comparaison de ses valeures.
2 objets sont donc égaaux quand ils ont la même valeur

Objet Vs Value Object

* 2 objets sont égaux quand ils référence la même zone mémoire.
* Une égalité peut éventuellement être faite sur un identifiant
* Object value se concentre uniquement sur les valeurs
Object value est immutable (non modifiable)

Intérêts

// Avec:

Les patterns comportementaux

Iterator

Problématique

Un objet complexe doit être traversable sans exposer sa représentation.
Traversable = itérable = je peux faire une boucle.
Les exemples les plus connus : List , Vector , HashMap, etc ...

Quand l'utiliser ?

  • Besoin de parcourir des objets
  • Besoin d'un moyen homogène d'itération entre plusieurs objets
Homogène = tous les itérateurs on la même interface.
Très souvent on ne créer pas d'itérateur , on utilse ceux mis à notre disposition.

Un peu de théorie

  • Aggregate : définit l'interface d'un objet composé permettant de créer un Iterateur.
  • ConcreteAggregate : sous-classe de l'interface Aggregate. Elle est composée d'éléments et implémente la méthode de création d'un Iterateur.
De prime abord Aggregate peut sembler superflue => utile si plusieurs classes
  • Iterator : définit l'interface de l'itérateur, qui permet d'accéder aux éléments de l'objet Aggregate.
  • ConcreteIterator : fournit une implémentation permettant de parcourir les éléments de ConcreteAggregate. Elle conserve la trace de la position courante.

Strategy

Problématique

Pouvoir choisir dynamiquement un algorithme
En d'autre terme réaliser différentes opérations avec un seul et même objet.

Quand l'utiliser ?

  • Plusieurs algorithmes similaires
  • Limitation des responsabilités et inversion de dépendance
* La même action de manière différente
* on évite d'ajouter des responsabilité en ajoutant méthodes à une classe. On dépend de l'interface et non des classes concrete.

Respect de SOLID :
* (Single responsability,
* Open close (ouvert à l'extension, fermer à la modif),
* Liskov substitution (pouvoir remplacer une instance par un type dérivé);
* Interface segregation : préféré plusieurs interface spécifique à une seule grosse
* Dependency Inversion.

Exemple : Un traitement d'image, choisi un algorithme en fonction d'un critère : rapidité ou qualité du résultat.

Un peu de théorie

* Une interface strategy qui défini le comportement * X implémentation concrète. * Une classe qui utilise la stratégie qu'on lui donne.

Template method

Problématique

Définir le squelette d'un algorithme en déléguant certaines étapes à des sous-classes.
Probablement un des des DP les plus simples.
On l'implémente souvent sans le savoir.

Quand l'utiliser ?

  • Un fonctionnement global commun
  • Des particularités à gérer par des sous classes
Exemple : Génération d'un document :
- Besoin d'une entête, d'un corps , d'un pied de page. => fonctionnement global
- Le contenu de ces section differt selon le type de doc => particularités

Un peu de théorie

On est très proche de STRATEGY
* Template est à utiliser quand l'algoritme ne change pas (juste son implémentation).
* Quand le choix de l'implémentation est fait à la compilation et non dynamiquement

Un classe abstraite défini l'algorithme.
Des méthodes abstraites
Une méthode "template" qui appel les methodes abstraites
On implémente ensuite les méthodes abstraites.

State

Problématique

Changer le comportement d'un objet en selon son état interne.
C'est un design pattern dérivé des machines à état.
Machine à état assez courant en informatique embarquée / electronique.
Dans le JV aussi , bonne alternative au réseau de neurone pour la prise de décision.

Quand l'utiliser ?

  • Isolation des algo propre à chaque état
  • Modification de l'instance et non recréation
  • Masquage de la complexité
State peut être replacer par if/else => pas éléguant et lourd si beaucoup d'état
Recréation d'objet (avec param par ex) => Pas efficient et pas éléguant.

Un peu de théorie

* Un interface d'état définissant les actions possible
* Des implémentations d'état
* Un objet utilisant un état.
Quand on change l'état l'implémentation change et donc le comportement.

Observer

Problématique

Notifier un ensemble d'objet d'un changement d'état
Par exemple réponse à un événemnt click.

Quand l'utiliser ?

  • Besoin d'informer un ou plusieurs objet immédiatement
  • Limiter le couplage
Couplage limité = L'objet observé n'est pas responsable de ce qui passe dans les objets observant.

Utiliser l'existant

  • Java : Observable
  • C# : IObservable
  • C++ : boost::signals2 ou QT signal/slot
  • PHP : SplSubject
  • ...

Un peu de théorie

Observer : Interface définissant la fonction que devra appeller l'objet observable.
ObserverA/B : Implémentation concrete
Subjet (observable) : Peut être une interface (cf celle des langages). Contient les observers et les gères.

Mediator

Problématique

Gérer l'intéraction de différents objets entre eux en limitant leur couplage.
Permettre à des objets d'avoir un effet les uns sur les autres tout en restant indépendant.
==> Réutilisabilité des objets

Quand l'utiliser ?

  • Ensemble d'objet devant intéragir ensemble
  • Pas de couplage entre ces objets
  • Pouvoir varier l'intéraction des objets indépendamment
DP couramment utilisé dans les interface graphique :
Le clic sur un élément d'UI en fait apparaitre ou disparaitre un autre.

Un peu de théorie

Collegue : définit l'interface d'un collègue. Il s'agit d'une famille d'objets qui s'ignorent entre eux mais qui doivent se transmettre des informations.
ConcreteColleagueA et ConcreteColleagueB : sont des sous-classes concrètes de l'interface Collegue. Elles ont une référence sur un objet Mediateur auquel elles transmettront les informations.
Mediator : définit l'interface de communication entre les objets Colleague.
ConcreteMediateur : implémente la communication et maintient une référence sur les objets Collegue.
/!\ Attention aux référence circulaire qui pourrait intervenir.

UI Mediator

Exemple d'élément d'une UI qui interagisse entre eux sans pour autant être couplé.
ON peut tout à fait modifier le comportement juste en changeant le mediateur.
