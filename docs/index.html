Principes des design patterns

Définition

Solution à un problème récurrent

Orienté objet

Indépendant du langage

Décrit une solution éprouvée à un problème récurrent.
Ex : Comment ajouter des fonctions à une classes sans la modifier
Ex : Comment garantir l'unicité d'instance d'une classe

Description

  • Nom
  • Problématique
  • Solution
  • Conséquences
Un pattern se décrit avec 4 éléments :
Nom : Permet de l'identifier et éventuellement de décrire le problème rapidement
Problématique : Une description du problème que le pattern résoud
Solution : Comment résoudre le problème. Souvent détaillé avec de l'UML
Conséquences : Quels sont les avantages et inconvenients de la solution

Quand ? Pourquoi ?

  • Dès qu'un problème de conception se pose
  • Utilisation d'une solution éprouvée
  • Productivité
  • Qualité de code
  • Bonnes pratiques
  • Over engineering
  • Penser aux Dp dès qu'on est face à un problème de conception
  • Permet d'utiliser une solution sure et éprouvée qui résoud le problème
  • Gain de productivité évident (code déjà fait , connu de tous)
  • Favorise les bonnes pratiques (couplage faible, lisibilité, evolutivité,etc ...)
  • Attention à ne pas en abuser. Peut cacher un problème de conception plus grave. Il préférable de refactorer vers les designs patterns si besoin

Répartition

  • Patrons créateurs
  • Patrons structuraux
  • Patrons comportementaux
Créateurs : Comment son créer les objets. => La famille la plus importante
Structuraux : Comment les classes sont organisée pour former un plus grand ensemble.
Comportementaux : Comment les objets communiquent entre eux et partage les responsabilité sur la gestion algorithmique.

Structuraux et comportementaux sont moins courants => plus souvent rencontré dans le dév de lib ou FW

Les patterns createurs

Singleton

Le plus connu mais aussi le plus controversé. Utilisation très discutable notamment selon les langages

Problématique

  • Une seule et unique instance
  • Accès à cette instance
2 objectifs :
* Une seule instance d'une classe
Important sur gros projets à plusieurs dev => on ne connais pas tout
* Offrir un moyen d'accéder à cette instance

Quand l'utiliser ?

Une seule instance d'une classe peut exister

  • Ressource système
  • Class d'application
  • Gestion de journaux *
* Port série
*
* Fait partie des utilisations très discutable. Des méthodes modernes permettent de s'en passer (DI)

Conséquences

  • Controle l'unicité d'instance d'une classe
  • Induit des problèmes de thread safety
Dans un environnement multi threadé , on risque de casser l'unicité d'instance. Voir plus tard exemple

La controverse

  • Utilité parfois discutable selon le langage
  • Signe de mauvais design
  • Difficile à tester unitairement
* Ok dans les environnement à mémoire partagée. Pas en PHP => instance propre à chaque requête , singleton peut ou pas utile. Suffit de pas instancier 2x la même classe.
* Une sur utilisation du singleton est souvent un signe de mauvais design. On préfère en général l'injection de dépendance.
Une seconde instance va elle crasher mon application => Si non singleton probablement pas nécessaire.
* Créer un couplace fort à l'environnement globale de l'application => rend les tests unitaires plus difficile.

Un peu de théorie

Implémentation naïve

						
public class Singleton {
	private static Singleton instance;

	private Singleton() {}

	public static Singleton getInstance() {
		if (instance == null) {
			instance = new Singleton();
		}
		return instance;
	}
}
- Implémentation fonctionnelle
- Problème de thread safety : - Environnement fortement multi threadé. Peu probable avec 2/3 threads (mais possible)
- plusieurs thread peuvent passer le if en même temps et créer plusieurs instance.

Implémentation Thread Safe

							
public class Singleton {
	private static Singleton instance = new Singleton();

	private Singleton() {}

	public static Singleton getInstance() {
		return instance;
	}
}
On délègue la synchronisation au class loader. Il le gère en interne.
Synchronisation de getInstance possible => /!\ performance
Solution peuvent être différente selon les langages.

Abstract Factory

Problématique

Fournir différents objets d'une même famille sans connaitre leur implémentation concrète

Quand l'utiliser ?

  • Objets regroupés en famille (UI , Documents)
  • Eviter les couplages forts
  • Masquer la complexiter d'instanciation
* Ui : Famille windows/linux; Document : Style moderne/classique * Pas de couplage vers les objets concret. * On expose uniquement les interfaces nécessaires

Fabrique non abstraite

  • Attention à faire la différence. Fabrique = bonne pratique. Pas un DP
  • Abstract factory permet une évolutivité plus simple
* Beaucoup moins de code à changer si nouvelle famille. * Une factory classique va demander plus de modification sur le code qui utilise la factory. * Factory => code figé , Abstract Factory => Code évolutif * Perso : Jamais utilisé d'abstract factory.

Builder

Problématique

Permettre d'obtenir des réprésentations différentes d'un objet avec le même procédé de construction

Séparer la construction d'un objet complexe de sa représentation. On déplace la logique de construction en dehors de l'objet à instancier.

Quand l'utiliser ?

  • Données de construction incomplète
  • Simplification de classe : limiter les constructeurs
* Exemple : parsage d'un document ou réception de données au fur et à mesure. * On évite de surcharger la classe de l'objet avec des infos liées à sa représentation. Ex : des if/else dans le constructeur pour initialiser l'objet. Ou encore plusieurs constructeurs

Un peu de théorie

Présenté pour le principe, mais rarement implémenté sous cette forme trop contraignante.

Solution à 

					
Pizza p = new Pizza();
p.setTaille(Taille.GRANDE);
p.setFromage(Fromage.CHEDDAR);
Ou encore 
					
						Pizza sansolive = new Pizza(Taille.GRANDE,Fromage.CHEDDAR,false);
						Pizza avecjambon = new Pizza(Taille.GRANDE,Fromage.CHEDDAR,false,true);
1- c'est pas très élégant et fastidieux à écrire. Problème si tous les param pas dispo au moment de la construction 2- Compliquer à gérer

Forme la plus courante

						
Pizza pizza = new PizzaBuilder()
	.setTaille(Taille.GRANDE)
	.setFromage(Fromage.MOZZA)
	.setJambon(true)
	.setOlive(false)
	.build();

Les patterns structuraux

Adapter

Problématique

Faire fonctionner ensemble des classes aux interfaces incompatible.
Exemple : Un driver fournis une interface, mais est incompatible avec le besoin du logiciel.
==> On doit adapter son comportement.

Quand l'utiliser ?

  • Besoin de convertir l'interface d'une classe en quelque chose de compréhensible
  • Impossible de modifier l'interface incompatible
Si c'est possible on préfèrera toujours adapter le code.
Exemple : API pilotage => multi caméra mais API constructeur imposé. Il faut adapter la notre.

Decorator

( a.k.a Wrapper )

Problématique

Ajouter des fonctionnalités à une classe dynamiquement.
DYnamique = à l'execution.

Quand l'utiliser ?

  • Ajout de responsabilités optionelles à un objet
  • Pas d'héritage
Responsabilité optionnelle :
=> On doit pouvoir choisir si on utilise ou non les nouvelles responsabilité.

Pas d'héritage :
* Evite l'héritage fonctionnel.
* Manque de souplesse => chaine d'héritage A => B => C , si je veux C je dois tout récupérer.

Exemple concret = la gestion des stream java/C# FileWriter,bufferedWriter , etc ...

Un peu de théorie

  • Component est une interface qu’implémentent à la fois l’objet décoré et les objets qui le décorent
  • ConcreteComponent représente l’objet décoré (qui implémente Component)
  • Decorator représente l’interface des décorateurs
  • On trouve ensuite les décorateurs concrets (ConcreteDecoratorA, etc.)
- L'objet concret et son décorateur hérite de la même interface.
- L'agregation indique qu'un décorateur peut contenir un "component"
- Principale problème => Le type de l'objet sera celui du dernier décorateur utilisé => peut être problématique à gérer.

Pratique

Exemple JAVA dispo
Le but :
* Une classe "Moderator" qui censure du text en remplacant certains mots.
* On ajoute au fur et à mesure la possibilité de censurer un numéro de téléphone puis un email.
La classe Decorator doit être abstraite.

Composite

Problématique

Gérer une hiérarchie d'objet sans limite de profondeur.

Quand l'utiliser ?

  • Hierarchie à profondeur indéterminée
  • Besoin de traiter un parent ou un enfant de manière identique
Chaque élément est un composant potentiel. En plus des éléments classiques, il y a un élément composite qui peut être composé de plusieurs composants. Comme l'élément composite est un composant potentiel, il peut être composé d'autres éléments composites.
L'exemple le plus concret => un système de fichier avec dossier/fichier (chacun peut être cliquer).

Un peu de théorie

Pattern assez naturel , on l'utilise assez souvent sans le savoir.

Facade

Vous l'avez déjà probablement déjà tous utilisé sans le savoir.

Problématique

Simplifier un sous système trop complexe.
Sous système = une partie/module d'un soft
Une API ou une librairie trop complexe

Quand l'utiliser ?

  • Simplifier la lecture/utilisation d'un sous système
  • Refactoring impossible => rétro compatibilité
  • Limiter les dépendances entre le client et le sous système
/!\ A ne pas mélanger avec l'adapteur qui ,lui, doit respecter une interface bien particulière
Facade = vraie volonté de simplification.

Un peu de théorie

Ex : Class A et B chacune une méthode. La facade masque la complexité des 2 méthodes.

En pratique

/!\ EX DE CODE APRES :
Site de voyage veux simplifier son API pour réserver un voyage complet.
Ne veux cependant pas perdre la possiblité des réservations unique

En pratique

									
Flight f = new Flight();
f.bookTicket();

t.bookCar();
